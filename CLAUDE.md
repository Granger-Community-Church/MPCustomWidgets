# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` — webpack watch build (development).
- `npm run build` / `npm run prod` — production build into `dist/`.
- `npm run create-widget` — interactive CLI scaffolding a new folder under `Widgets/` from `Widgets/WidgetTemplate/`. See `cli/create-widget.md`.
- `npm run cdn-url` — print jsDelivr CDN URLs for the current branch's `dist/` artifacts.
- `npm run make-cloud-widget` — convert a widget's local `demo.html` to `cloud-demo.html` by rewriting paths to jsDelivr CDN URLs (so the demo can be served from a CDN-hosted page).

There is no test suite, linter task, or formatter in `package.json`. ESLint is a devDependency but not wired to a script.

## Big-picture architecture

This repo is two things in one:

1. **A small JS runtime** (`src/`) that mounts on any page and renders MinistryPlatform-backed widgets defined declaratively via `data-*` attributes.
2. **A library of example widgets** (`Widgets/<Name>/`), each a self-contained folder of demo HTML + Liquid template + supporting SQL/assets. Widgets are independent — there is no shared widget framework beyond the data-attribute contract.

### Runtime flow (`src/`)

- `src/index.js` — entry. On `window load`, calls `WidgetModule.Init()` and exposes `window.ReInitWidget` / `window.ReinitAllWidgets`.
- `src/Modules/WidgetModule.js` — the only module. Finds every `[data-component="CustomWidget"]`, validates its attributes, optionally pulls a JWT from `localStorage` (`mpp-widgets_AuthToken` / `mpp-widgets_ExpiresAfter`), then calls `LoadWidget` which:
  1. Resolves the template via priority: `data-templateId` (inline `<script type="text/template">`) → `data-template` (local path) → `data-templateUrl` (HTTPS, allowlisted via `TEMPLATE_URL_ALLOWLIST`).
  2. Calls either the MP REST stored-procedure endpoint (default) or the MP Events API (`data-useCalendar="true"`, deprecated).
  3. If the stored-procedure name ends in `_json`, parses the single `JsonResult` column instead of the multi-DataSet shape.
  4. Renders with LiquidJS and dispatches a `widgetLoaded` CustomEvent on the element and `window`.
- `src/Services/APIService.js` — bare `fetch` wrappers; no MP-specific logic lives here.
- `src/Services/TemplateService.js` — wraps LiquidJS, registers custom filters (e.g. `mp_currency`).

### Build output (`dist/`)

Webpack bundles `src/index.js` → `dist/js/customWidgetV1.js` (the production filename never changes — it's the public CDN URL). `dist/css/mp-custom.css` and `dist/js/forceLogin.js` are copied through unmodified by `CopyWebpackPlugin`. `dist/` IS committed because consumers load it directly from jsDelivr at `cdn.jsdelivr.net/gh/MinistryPlatform-Community/MPCustomWidgets@main/dist/...`.

`drop_console: false` is intentional in `webpack.config.js` — widget runtime logs are part of the supported debugging UX (`data-debug="true"`).

### Widget folder convention

A widget under `Widgets/<Name>/` typically contains:
- `demo.html` — local-paths demo (loads `/dist/js/customWidgetV1.js` and `/Widgets/<Name>/Template/...`).
- `cloud-demo.html` — generated by `make-cloud-widget`; same demo with all paths rewritten to jsDelivr.
- `Template/<Name>.html` — LiquidJS template.
- `StoredProc/*.sql` — the `api_Custom_*` stored procedures the widget calls (informational; deploying them to MP is manual).
- `Schema/`, `Assets/`, optional `<Name>.js` for widget-side interactivity.

Widgets are not built or registered anywhere; they are just files served from this repo via jsDelivr. Adding a new widget = create the folder; it does not need to be wired into the runtime.

## Conventions worth knowing

- **CDN-first distribution.** Consumers reference `@main` on jsDelivr; we tolerate jsDelivr's cache lag and purge manually rather than version-pinning.
- **Shared CSS lives in a separate repo** (`Granger-Community-Church/MP-Widgets-CSS`, served via jsDelivr). `src/css/mp-custom.css` is the *upstream community* stylesheet; Granger-specific demos pull from the separate `MP-Widgets-CSS` repo.
- **MP API datetimes are local time** despite carrying a `Z` suffix. Never construct a JS `Date` from them as UTC — parse as local. Use `dbo.dp_DomainTime()` for "now" in MP views/SP filter clauses.
- **Stored procedure contract.** Names must start with `api_Custom_` and be registered in MP metadata. Required params: `@DomainID int`, and (almost always) `@Username nvarchar(75) = null`. JSON-mode SPs must end with `_json` and return one row, one column named `JsonResult`.
- **`data-host` is the church prefix only** (e.g. `grangerchurch`, not a URL). The runtime rejects values containing `.` or `http`.
- **Template URL allowlist.** Remote templates (`data-templateUrl`) are restricted by regex in `WidgetModule.TEMPLATE_URL_ALLOWLIST` (jsDelivr, `*.github.io`, raw.githubusercontent.com). Modify that array to add domains.
- **Client-side fuzzy search uses Fuse.js.** Pinned to a specific minor (e.g. `fuse.js@7.0.0`) and loaded from jsDelivr. `<script>` tags inside Liquid templates won't execute — templates render via `innerHTML` — so the widget JS injects the loader into `document.head` itself and gates search binding on a Promise. Stop words and synonyms are pre-applied in app code rather than via library term-processor hooks (MiniSearch's `processTerm` proved fragile during the survey — `campus` leaked into the index). Reference implementation: `Widgets/GrangerEvents/GrangerEvents.js`. Side-by-side comparison harness for re-evaluating against alternatives: `Widgets/GrangerEvents/compare-search.html`.

## Notes on the upstream vs. fork

`package.json` ships as `MinistryPlatform-Community/MPCustomWidgets`, but the local `Widgets/Granger*` folders, `MP-Widgets-CSS` styling, and `Granger-Community-Church/MPCustomWidgets` CDN references in some demos indicate this checkout is the Granger fork. When in doubt about which org a CDN URL should target, check the surrounding file's existing URLs rather than assuming.
